## 1. 概述

本文档规定 AutoPrint 仪表盘每次上线新版本时的**版本记录规范**，确保团队成员能一致地追溯"这个版本改了什么、何时发布、由谁发布"。

版本信息统一存放在 **Supabase `app_versions` 表**，并在仪表盘「📋 版本记录」Tab 中可视化展示。不依赖 GitHub Releases（纯 Web 应用无下载产物，Releases 不适用）。

  ------------------------------------------------------------------
  字段              类型              说明
  ---------------  ----------------  ----------------------------------
  version          TEXT (唯一)       语义化版本号，如 `1.2.0`
  released_at      TIMESTAMPTZ       发布日期时间（含时区）
  changes_summary  TEXT              变更摘要（一句话到一段，中文）
  commit_hash      TEXT (可空)       关联 git commit 短哈希，如 `3324419`
  deployed_by      TEXT (可空)       发布人
  created_at       TIMESTAMPTZ       记录创建时间（系统自动）
  ------------------------------------------------------------------

------------------------------------------------------------------------

## 2. 版本号规则（语义化版本）

采用 `主版本.次版本.修订号` 三段式：

  ------------------------------------------------------------------
  变更类型          触发场景                         示例
  ---------------  --------------------------------  ---------------
  修订号 +1         缺陷修复、文案修正、小调整       `1.2.0` → `1.2.1`
  次版本 +1         新功能、新 Tab、可见增强         `1.2.0` → `1.3.0`
  主版本 +1         重大重构、架构变更、breaking     `1.2.0` → `2.0.0`
  ------------------------------------------------------------------

- 初始上线为 `1.0.0`。
- 版本号**一经发布不可修改**（`version` 字段设了 UNIQUE 约束）。写错只能新增一条修正记录。

------------------------------------------------------------------------

## 3. 发布触发条件

满足以下任一情况即应发布并记录一个版本：

- 新功能合并到 `master` 并部署到 Vercel 生产环境
- 修复了影响用户使用的 Bug
- 重大 UI/UX 调整、品牌规范更新
- 数据表结构变更（如新增/调整 Supabase 表）

**纯本地调试、未部署到生产**的改动**不需要**发版本。

  ------------------------------------------------------------------
  ☐  代码已本地 Go Live 测试通过，无控制台报错

## 5. 记录版本（二选一）

### 方式 A：仪表盘「📋 版本记录」Tab（推荐，日常用）

1. 用 **admin 账号**登录仪表盘。
2. 切到「📋 版本记录」Tab，点击右上角 **➕ 记录新版本**。
3. 填写弹窗：
   - **版本号**：如 `1.3.0`
   - **发布日期**：默认今天，可改
   - **变更摘要**：本次改了什么（参考第 6 节）
   - **Commit 哈希**：粘贴 `git rev-parse --short HEAD` 的输出
   - **发布人**：你的名字
4. 保存 → 列表即时刷新，全员可见。

> 非 admin 账号看不到「➕ 记录新版本」按钮（受 Supabase RLS 限制：登录用户可读，仅授权角色可插入）。

### 方式 B：SQL 直插（批量 / 补录历史用）

```sql
-- 在 Supabase SQL Editor 执行
INSERT INTO public.app_versions (version, released_at, changes_summary, commit_hash, deployed_by)
VALUES (
  '1.3.0',
  NOW(),
  '本次变更摘要（中文，一句话到一段）',
  '3324419',          -- git 短哈希
  'Eric Zhang'
);
```

> 复现/回滚脚本见仓库 `scripts/create_app_versions.sql`（含建表、RLS、初始种子）。

## 6. 变更摘要写法

- **简洁、面向团队**：说明"做了什么 + 为什么"，不写实现细节。
- 关联 Issue 时带上编号，如 `（Issue #21）`。
- 多条变更用顿号或分号分隔，单条不要超过一行半。

**好例子**：
> 新增「年度负载高峰」Tab：按日/周/月统计当前自然年打印峰值 + 月度图表（Issue #18）。

**差例子**：
> 改了 index.html，加了一个函数，调了 supabase.from，修了 bug。

------------------------------------------------------------------------

## 7. 与 GitHub Issue 的协作

- 每个功能/修复对应一个 GitHub Issue。
- 提交时 message 引用 Issue：`feat: ... (Issue #21)`。
- 发布并记录版本后，**关闭对应 Issue**。
- 版本记录里的 `commit_hash` 可反向定位代码，形成"Issue → Commit → 版本"可追溯链。

------------------------------------------------------------------------

## 8. 示例：一次完整发布流程

```
1) 开发完成 → 本地 Go Live 测试通过
2) git add index.html
3) git commit -m "feat: XXX 功能 (Issue #23) ✅ 已在本地环境测试通过"
4) git push origin master
5) 访问 Vercel 线上 URL，确认功能正常
6) git rev-parse --short HEAD   →  得到 e.g. a1b2c3d
7) 仪表盘用 admin 登录 → 「版本记录」Tab → ➕ 记录新版本
   版本号: 1.3.0
   摘要:   XXX 功能上线（Issue #23）
   Commit: a1b2c3d
   发布人: Eric Zhang
8) 关闭 GitHub Issue #23
```

------------------------------------------------------------------------

## 9. 注意事项

- 版本号写错不要改历史，新增一条修正记录并在摘要注明。
- `commit_hash` 尽量填，便于回溯；不填也允许。
- 历史里程碑已预置：`1.0.0`(07-03) / `1.1.0`(07-08) / `1.2.0`(07-09)。
- 如需"对外正式发布"能力，未来可叠加 GitHub Releases（打 tag `v1.x.x`），通过 `commit_hash` 与 `app_versions` 关联，本期不强制。
