## 一、执行摘要

### 核心问题
使用 `comlan-ppt-brand` Skill 生成 PPT 时，表格列宽溢出问题反复出现，经历了 **6 个版本** 才彻底解决。

### 根本原因
**pptxgenjs 库的 `addTable()` API 对中文多列表格完全不可靠**，且缺乏统一的安全边界约束机制。

### 最终方案
1. **彻底放弃 `addTable()`**，改用手绘表格（shape + text）
2. **引入 SAFE 常量对象**，统一所有横向元素的安全边界
3. **百分比列宽 + 极度保守参数**（totalW≤9.5", 字体≤9pt, 边框≤0.5pt）

## 三、根本原因分析（Root Cause Analysis）

### 3.1 技术根因

#### pptxgenjs 的 addTable() API 缺陷
```
问题描述：addTable() 的 colW 参数在当前版本完全无效
影响范围：所有使用 addTable() 的表格
验证方法：诊断文件（3个表格列宽完全相同）
结论：必须用 shape + text 手绘表格
```

#### 中文文字宽度计算问题
```
问题描述：pptxgenjs 无法正确计算中文文字宽度
表现：指定 w=2" 但实际渲染可能占用 2.5"
叠加效应：边框 + padding + 字体渲染误差 → 累积溢出
解决方案：totalW 必须远小于幻灯片宽度（9.5" vs 13.333"）
```

#### 缺乏统一安全约束
```
问题：表格用 9.5" 安全，但线条用 12.5"
结果：表格不溢但线条溢（v1.5 问题）
解决：引入 SAFE 常量对象，统一所有元素边界
```

### 3.2 流程根因

#### 假设未验证
- 假设 `colW` 参数有效 → 浪费 2 个版本（v1.1-1.2）
- 假设 `wrap:true` 能解决溢出 → 浪费 1 个版本（v1.3）
- 假设手绘表格的绝对值 colWidths 精确 → 浪费 1 个版本（v1.4）

#### 缺乏 diagnostics 意识
- 应该在第一版就创建诊断文件，验证 colW 是否有效
- 应该在一开始就测试"边界情况"（最长文字 + 最多列数）

## 四、关键技术经验

### 4.1 ✅ 有效方案

#### 1. 手绘表格（shape + text）
```javascript
// 核心函数：drawTable()
// 优势：100% 精确控制列宽
// 劣势：代码量大，需要手动计算位置

function drawTable(slide, x, y, totalW, colHeaders, dataRows, opts) {
    // 用 addShape('rect') 画单元格背景
    // 用 addText() 在绝对位置写文字
    // 列宽用百分比分配，避免绝对值误差
}
```

#### 2. 百分比列宽（colPercents）
```javascript
// ✅ 推荐：百分比
colPercents: [18, 17, 38, 27]  // 总和=100

// ❌ 避免：绝对值
colWidths: [1.8, 1.7, 3.8, 2.7]  // 容易计算误差
```

#### 3. SAFE 常量对象
```javascript
const SAFE = {
    x: 0.4,           // 左边距
    maxW: 9.5,        // 最大内容宽度
    rightBound: 9.9   // 绝对右边界
};
```

### 4.2 ❌ 无效方案

#### 1. addTable() + colW
```javascript
// ❌ 无效：colW 参数完全被忽略
slide.addTable(rows, { x: 0.5, y: 1.3, w: 12, colW: [2, 3, 4, 3] });
```

#### 2. addTable() + wrap:true
```javascript
// ❌ 无效：wrap 对中文支持差，且无法解决列宽问题
slide.addTable(rows, { w: 11, wrap: true });
```

#### 3. 缩短文字 + 缩小字体
```javascript
// ❌ 治标不治本：文字再短，列宽错误仍然溢出
// 只在手绘表格 + 安全参数基础上作为辅助手段
```

## 五、设计纪律（Design Discipline）

### 5.1 安全参数约束

**必须遵守的硬性规则**（经 v1.0-1.6 验证）：

| 参数 | 最大值 | 说明 |
|------|--------|------|
| totalW | 9.5" | 表格总宽度（幻灯片宽 13.333"） |
| 字体 | 9pt | 表格内文字（标题可 22pt） |
| 边框 | 0.5pt | 表格边框（1pt 太粗） |
| paddingX | 0.06" | 单元格左右 padding |
| paddingY | 0.01" | 单元格上下 padding |
| 列数 | 4 | 超过 4 列需要更小字体 |

### 5.2 代码审查清单

**生成 PPT 前必须核对**：

- [ ] 是否使用了 `addTable()`？（如果是，改为 `drawTable()`）
- [ ] 所有横向元素是否遵守 `SAFE.maxW`？
- [ ] 表格列宽是否用百分比（colPercents）？
- [ ] 是否有中文字符超出单元格？（测试最长可能的文字）
- [ ] 页码位置是否在 `SAFE.rightBound` 内？

## 九、总结

### 关键洞察
1. **不要假设第三方库 API 行为**（必须验证）
2. **中文排版需要更保守的参数**（经验值：比英文小 20-30%）
3. **统一约束 > 局部优化**（SAFE 常量对象的价值）
4. **手绘 > 声明式**（当需要精确控制时）
